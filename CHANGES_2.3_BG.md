# Wellness Portal 2.3 — поправка на йерархията и ролите

## Какво е променено

- В раздел **„Структура / Йерархия“** вече има бутон `⇄` за директна смяна на роля:
  - треньор → клиент;
  - клиентски акаунт → треньор.
- Собственият акаунт не може да променя сам ролята си.
- След смяна на роля структурата се зарежда отново и акаунтът се показва само в новата си роля.
- Един акаунт се визуализира само веднъж, дори ако в стара база има повече от един свързан `clients` ред.
- Добавена е допълнителна защита срещу показване на стар несвързан „сенчест“ клиентски запис, създал дублиране след `migration_10.sql`.
- Телефонът и потребителското име остават видими до името в йерархията.

## Миграция 11

Файлът `supabase/migration_11.sql`:

- намира дублирани клиентски записи със същия `user_id`;
- избира най-пълния запис и запазва него като основен;
- прехвърля всички измервания от останалите записи;
- запълва липсващи телефон, адрес, имейл, снимка, рождена дата, височина и бележки;
- премахва само обединените дубликати;
- разпознава безопасно стар минимален linked запис и по-стар несвързан запис със същото име и собственик;
- гарантира уникален `clients.user_id` чрез уникален индекс;
- предоставя защитена функция, която Edge Function използва преди всяка смяна на роля.

## Задължително обновяване

1. Направете backup на Supabase базата.
2. Изпълнете в SQL Editor:

```text
supabase/migration_11.sql
```

3. Публикувайте отново Edge Function-а:

```bash
supabase functions deploy manage-account --no-verify-jwt
```

4. Публикувайте новия frontend:

```bash
npm install
npm run build
```

Миграцията е проектирана да запази клиентската информация и историята на измерванията. При конфликт за еднакъв параметър на еднаква дата се запазва наличната непразна стойност в основния запис.
